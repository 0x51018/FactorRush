# FactorRush 코드베이스 가이드

## 1. 프로젝트 개요

FactorRush는 로그인이나 DB 없이, 초대 링크만으로 방을 만들어 함께 플레이하는 숫자 게임 프로토타입입니다.  
현재는 다음 두 모드를 지원합니다.

- 소인수분해 레이스
- 10진수/2진수 변환 레이스

핵심 목표는 "빠르게 플레이 가능한 엔드투엔드 구조"를 먼저 만드는 것이었고,  
세부 게임 규칙은 이후 반복적으로 수정할 수 있도록 서버 메모리 기반 상태 머신 구조로 잡았습니다.

## 2. 모노레포 구조

```text
apps/
  server/
    src/index.ts
    src/roomStore.ts
  web/
    src/app/layout.tsx
    src/app/page.tsx
    src/app/room/[roomId]/page.tsx
    src/app/globals.css
    src/components/game-shell.tsx
    src/components/game-shell.module.css
packages/
  shared/
    src/index.ts
```

## 3. 패키지별 역할

### `apps/server`

서버 앱입니다.

- HTTP 서버 실행
- Socket.IO 연결 수락
- 방 생성/입장/재접속 처리
- 플레이어/관전자 분리 처리
- 라운드 시작/종료/점수 반영
- 최종적으로 모든 클라이언트에 현재 방 상태 브로드캐스트

중요한 점은, 서버가 "정답 판정과 점수 반영의 기준점"이라는 점입니다.

### `apps/web`

Next.js App Router 기반 브라우저 클라이언트입니다.

- 홈 페이지(`/`)
- 방 진입 페이지(`/room/[roomId]`)
- 클라이언트 셸 기반 실시간 UI
- 로비 설정 화면
- 라운드 진행 화면
- 점수판 / 결과 화면

Next 라우팅은 `src/app`이 담당하고, 실제 실시간 상태와 UI는 `game-shell.tsx`가 담당합니다.

### `packages/shared`

서버와 클라이언트가 함께 사용하는 계약 계층입니다.

- 타입 정의
- 소켓 ack 구조
- 방 스냅샷 구조
- 플레이어/관전자 역할 구조
- 문제 생성 로직
- 정답 정규화/검증 로직
- 점수 계산 로직

이 패키지가 있기 때문에 서버와 클라이언트가 같은 데이터 구조를 바라봅니다.

## 4. 파일별 설명

### 서버

#### `apps/server/src/index.ts`

서버 진입점입니다.

- Express 서버 생성
- Socket.IO 서버 생성
- 각 소켓 이벤트를 `RoomStore` 메서드에 연결

즉, 이 파일은 "연결점"이고 비즈니스 로직은 거의 없습니다.

#### `apps/server/src/roomStore.ts`

현재 프로젝트에서 가장 중요한 서버 파일입니다.

- 방 메모리 저장소 보관
- `socket.id -> room/member` 역추적
- 호스트 권한 관리
- 재접속 처리
- 플레이어/관전자 좌석 분리
- 라운드 타이머 관리
- 정답 검증 후 점수 반영
- 클라이언트 전송용 `RoomSnapshot` 생성

이 파일을 기준으로 현재 게임의 흐름이 결정됩니다.

### 클라이언트

#### `apps/web/src/app/layout.tsx`

Next 루트 레이아웃입니다.

- 전역 CSS 로드
- 메타데이터 정의

#### `apps/web/src/app/page.tsx`

홈 페이지 엔트리입니다.

- 게임 셸을 "홈 화면 모드"로 렌더링합니다.

#### `apps/web/src/app/room/[roomId]/page.tsx`

방 링크 전용 엔트리입니다.

- URL의 `roomId`를 받아 게임 셸에 넘깁니다.

#### `apps/web/src/components/game-shell.tsx`

현재 웹 앱의 메인 클라이언트 컨테이너입니다.

- Next 라우터 기반 홈/방 진입 분기
- Socket.IO 연결과 ack 처리
- 로컬 스토리지 기반 닉네임/재접속 처리
- 진행 중 방 입장 시 관전자 처리
- 로비 / 라운드 / 결과 화면 렌더링

현재는 한 파일에 모여 있지만, 다음처럼 나누기 좋습니다.

- `hooks/useRoomConnection.ts`
- `components/LandingView.tsx`
- `components/LobbyPanel.tsx`
- `components/RoundPanel.tsx`
- `components/Scoreboard.tsx`

#### `apps/web/src/app/globals.css`

전역 토큰과 기본 리셋 스타일 파일입니다.

#### `apps/web/src/components/game-shell.module.css`

게임 셸 전용 스타일 파일입니다.

- 메인 레이아웃
- 랜딩 포스터형 히어로
- 로비/라운드/리더보드
- 반응형 처리

### 공용

#### `packages/shared/src/index.ts`

서버와 클라이언트가 공통으로 쓰는 단일 계약 파일입니다.

- 타입
- 룸 상태 구조
- 입력 정규화
- 문제 생성
- 정답 검증
- 점수 계산

현재는 단일 파일이지만 다음처럼 분리 가능성이 높습니다.

- `types.ts`
- `challenge.ts`
- `validation.ts`
- `scoring.ts`
- `room.ts`

## 5. 모듈 관계

현재 의존 방향은 아래처럼 단순합니다.

```text
web ----\
         > shared
server --/
```

- `web`은 `shared`를 import 해서 타입과 유틸을 사용합니다.
- `server`도 `shared`를 import 해서 타입과 규칙 로직을 사용합니다.
- `web`과 `server`는 직접 서로를 import 하지 않습니다.

이 구조 덕분에 프론트/백 간 데이터 계약이 틀어질 가능성을 줄였습니다.

## 6. 런타임 흐름

### 현재 연결 방식

- 도커 컨테이너나 컨테이너 네트워크는 사용하지 않습니다.
- 개발 환경에서는 `Next.js 웹 서버(:3000)`와 `Socket 서버(:3001)`가 별도 프로세스로 실행됩니다.
- 프론트엔드는 `NEXT_PUBLIC_SERVER_URL`이 없으면 기본적으로 `http://localhost:3001`로 연결합니다.
- 서버와 프론트는 HTTP/소켓 레벨로만 연결되며, 별도 내부 브리지나 프록시는 두지 않았습니다.

### 방 생성

1. 클라이언트가 `room:create`를 보냅니다.
2. 서버가 방 코드와 호스트 플레이어를 생성합니다.
3. 서버가 소켓을 해당 room에 join 시킵니다.
4. 서버가 `room:state`를 브로드캐스트합니다.

### 방 입장

1. 클라이언트가 `room:join`을 보냅니다.
2. 서버가 room 존재 여부를 확인합니다.
3. 재접속 대상이 있으면 플레이어나 관전자 세션을 복구합니다.
4. 로비면 새 플레이어로 입장시키고, 게임 진행 중이면 관전자로 입장시킵니다.
5. 서버가 최신 room snapshot을 브로드캐스트합니다.

### 관전자 입장

- `round-active`, `round-ended`, `finished` 단계에서 새로 들어오는 사용자는 기본적으로 관전자입니다.
- 관전자는 채팅과 상태 열람은 가능하지만 정답 제출, 준비, 설정 변경, 방장 액션은 할 수 없습니다.
- 관전자는 로비로 돌아온 뒤 `플레이어로 참가` 버튼으로 좌석에 합류할 수 있습니다.
- 이 설계는 진행 중인 게임의 공정성을 유지하면서도 중도 합류를 허용하기 위한 절충안입니다.

### 라운드 진행

1. 호스트가 `game:start` 또는 `round:next`를 보냅니다.
2. 서버가 문제를 생성합니다.
3. 서버가 라운드 종료 타이머를 설정합니다.
4. 클라이언트는 `room:state`를 받아 라운드 화면을 그립니다.

### 정답 제출

1. 플레이어가 `round:submit-answer`를 보냅니다.
2. 서버가 `evaluateSubmission()`으로 정답 여부를 판정합니다.
3. 정답이면 순위와 점수를 계산합니다.
4. 모든 접속 플레이어가 정답을 제출했거나 시간이 끝나면 라운드를 종료합니다.
5. 서버가 다시 `room:state`를 브로드캐스트합니다.

## 7. 왜 서버가 정답 검증을 담당하는가

현재 모드만 보면 클라이언트 로컬에서도 정답 검증은 충분히 빠르게 가능합니다.  
그럼에도 서버에서 검증하도록 만든 이유는 다음과 같습니다.

- 점수 계산 기준을 서버 하나로 고정하기 위해
- 클라이언트 변조나 치트 가능성을 줄이기 위해
- 이후 모드가 더 복잡해져도 구조를 유지하기 위해
- "정답 판정 + 순위 확정 + 라운드 종료"를 한 트랜잭션처럼 다루기 위해

즉, 현재 성능 때문이 아니라 "상태 일관성" 때문에 서버 판정을 택한 것입니다.

## 8. 현재 구조의 장점과 한계

### 장점

- 단순함
- DB 없이도 즉시 플레이 가능
- 프론트/백 타입 공유
- Next 기반 페이지 구조 도입 가능
- 규칙 변경 지점이 비교적 명확함
- 진행 중 방 입장도 관전자 흐름으로 자연스럽게 흡수 가능

### 한계

- 서버 재시작 시 방이 사라짐
- 수평 확장 불가
- `game-shell.tsx`가 점점 커질 가능성
- `shared/src/index.ts`가 비대해질 가능성

## 9. 다음 리팩터링 추천 순서

1. 클라이언트 UI를 화면/패널 단위 컴포넌트로 분리
2. 소켓 연결 로직을 커스텀 훅으로 분리
3. shared 패키지를 규칙/타입/점수 모듈로 분리
4. 서버 룸 상태에 테스트 가능한 순수 함수 계층 추가
5. 실제 게임 규칙 확정 후 라운드 상태 머신 정교화

## 10. 현재 Next.js 전환 상태

지금 구조는 이미 `apps/web` 기준으로 Next.js App Router로 전환된 상태입니다.

- `packages/shared` 유지
- `apps/server` 유지
- `apps/web`만 Next.js로 교체

즉, 서버 도메인 로직은 거의 그대로 두고 프론트엔드만 교체한 구조입니다.
